# 策划立法：现状清单与当前问题

> 目的：把"知识库目前需要策划立法的所有部分"一次性摊开，让策划清楚**要维护什么、规模有多大**，
> 并集中说明当前的核心痛点——**规则过于复杂繁多、不够友好**，供我们一起讨论怎么精简。

---

## 一、什么叫"策划立法"

知识库在"知识构建 → 质量门禁 → 发布"整条流水线里，所有"什么算合格、什么能发布、对象关系怎么算数"
的标准，都由策划维护的一份**规则 Profile**决定。系统只负责把规则编译进流水线执行，**不替策划定标准**。

规则目前分成 7 大块（对应「策划立法」页的 7 个标签页）。下面逐块列出策划要填的东西和**当前默认规模**。

---

## 二、当前需要策划立法的全部内容（含数量）

| 模块 | 策划要维护什么 | 当前默认数量 |
|---|---|---|
| ① 文档类型 + Wiki 模板 | 每类资料成文时必须写清楚的章节/事实/证据要求 | **7 类**，合计约 **41 个必填章节 + 29 个必填事实** |
| ② 页面类型 + Wiki 标准（高级） | 与①几乎一一对应的另一套章节/事实标准 + 目录/模板文件 | **7 类**（与①重复，再来一遍 ~40 章节/~27 事实） |
| ③ 业务对象（实体类型） | 知识图谱里允许出现哪些对象 | **17 个** |
| ④ 业务关系（关系类型） | 对象之间允许什么关系、方向、是否自动采信、能否发布 | **16 个**，每个 3~4 个属性 |
| ⑤ 表字段推断策略 | 用字段名后缀区分"自动采信 / 必须人工确认" | **2 组后缀**（自动组默认 10 个后缀） |
| ⑥ 质量红线 | 哪些问题阻断发布 / 关注 / 仅记录 | **8 条** |
| ⑦ 版本管理 | Profile 命名、保存草稿/启用、历史版本、hash 比对 | 每次改动产生 1 个新版本 |

**粗略合计**：约 **47 个条目**（7+7+17+16）、**130+ 个"必填项"标签**、加上各类开关/方向/ID/后缀，
整份规则大约有 **300+ 个可调项**。这就是"繁多"的来源。

---

## 三、每块具体要填什么

### ① 文档类型 + Wiki 模板（7 类）
默认：系统规则、活动玩法、配置表说明、字段说明、数值规则、概念说明、操作流程。
每一类策划都要填：文档类型名称、用途说明、模板说明、**必填章节**（如"概览/核心规则/入口与条件/配置表依赖/数值与奖励/边界与异常/证据"）、**必填事实**（如 `system_name`、`entry_condition`、`config_table`、`source`）、是否必须有证据、是否允许发布，外加两个"高级标识"ID。

### ② 页面类型 + Wiki 标准（7 类，藏在"高级规则"页）
默认：system / activity / table / field / numeric / ui_flow / concept。
内容与①**高度重复**：同样的必填章节、必填事实、是否要证据、是否可发布，再加上 `目录(dir)`、`模板文件(template)`、`类型标识`。

### ③ 业务对象（17 个）
system、activity、config_table、field、resource、item、currency、reward、cost、condition、state、numeric_item、ui_element、progression、achievement、task、concept。每个填：对象 ID、业务名称、是否可发布。

### ④ 业务关系（16 个）
depends_on(依赖)、affects(影响)、contains(包含)、references(引用)、produces(产出)、consumes(消耗)、prerequisite_of(前置)、mutually_exclusive_with(互斥)、configured_in(配置于)、configured_by_field(由字段配置)、belongs_to(属于)、unlocks(解锁)、grants(授予)、costs(需要消耗)、has_field(拥有字段)、fk_to(外键指向)。每个填：关系 ID、业务名称、方向(A→B / 双向)、是否系统自动采信、是否可发布。

### ⑤ 表字段推断策略
- 可自动采信的字段后缀：`Id、Ids、TableId、ConfigId、RewardId、ItemId、CostId、ConditionId、StateId、ActivityId`
- 必须人工确认的字段后缀：（默认空）

### ⑥ 质量红线（8 条）
`required_wiki_sections_missing`(缺必填章节)、`required_facts_missing`(缺关键事实)、`source_trace_missing`(无法追溯来源)、`illegal_relation_type`(非法关系类型)、`concept_overuse`(概念滥用)、`candidate_relation_unconfirmed`(候选关系未确认)、`field_relation_unconfirmed`(字段关系未确认)、`stale_rule_profile`(规则版本过期)。每条填：规则 ID、等级(阻断/关注/仅记录)、说明、是否启用。

## 四、当前的核心问题（不友好的地方）

1. **同一套标准要维护两遍：文档类型 vs 页面类型重复。**
   ①和②都带"必填章节/必填事实/是否要证据/能否发布"，而且默认就是 7 对几乎一模一样的内容。
   改了一个不改另一个，标准就会对不上。这是最大的负担。

2. **数量过多、没有分组和默认继承。**
   47 个条目里很多是"每类都要重复写一遍概览/证据"。没有模板继承，策划只能不停复制粘贴样板。

3. **大量技术名词泄漏到策划界面。**
   `类型标识 / ID`、`目录(dir)`、`模板文件(template)`、`defaultPageTypeId / Wiki 类型 ID`、
   关系 `direction` 枚举、质量红线的英文 ID（`required_wiki_sections_missing` 等）——
   这些是工程概念，不是业务语言，策划很难判断该填什么、填错有什么后果。

4. **"必填事实"是机器键，不是业务提示。**
   形如 `system_name`、`entry_condition`、`config_table`、`reward_or_cost`，更像代码字段名。
   策划要自己造、还要保持各处一致，没有候选项、没有校验、没有示例。

5. **同一个意图散落在多处、耦合不直观。**
   "这类文档必须有证据"既在①的开关里设，又在⑥的 `source_trace_missing` 红线里设；
   "关系算不算数"由④的方向/自动采信 + ⑤的字段后缀**共同**决定。
   要改一个效果，往往得同时动好几个地方，且看不出它们的联动。

6. **关系/字段推断策略心智负担重。**
   16 种关系 + 后缀规则共同构成一套隐式的推断策略，到底什么会被自动采信、什么进候选审核，
   不了解引擎就判断不了。

7. **版本/hash/启用等概念是额外负担。**
   每次小改都要命名版本、选择"保存草稿/启用"、对比 hash，对只想"改一条标准"的策划偏重。

## 五、可讨论的简化方向（待和策划一起定）

> 这些只是候选思路，不是结论，列出来方便我们取舍。

- **合并①②为一层**：策划只维护"文档类型 + 必填章节/事实/证据"，目录/模板/页面类型 ID 全部由系统自动生成，策划不再看到第二套。
- **必填事实改成选择式**：把常用事实做成可勾选的业务清单（带中文名+示例），而不是手填机器键。
- **章节/事实模板化**：内置"概览/证据"等通用段落，新建文档类型时自动带上，策划只补差异部分。
- **质量红线收敛为几个开关**：把 8 条红线归并成策划能直接判断的 3~4 个业务选项（如"必须能追溯来源""关系必须人工确认"），隐藏英文规则 ID。
- **隐藏全部技术 ID 与版本机制**：日常编辑不暴露 ID/dir/template/hash；保存即生效，版本留痕放到后台。
- **给"最小可用规则集"**：先只保留团队真正会用到的少量文档类型/对象/关系，其余按需再加，而不是一上来 47 项。

---

### 一句话总结给策划
> 现在要立法的东西大约有 **7 类文档标准（还重复成两套）+ 17 个对象 + 16 种关系 + 字段后缀策略 + 8 条质量红线**，
> 合计三百多个可调项，且夹杂大量技术名词和重复维护点。
> 我们希望和策划一起把它**砍到一套、用业务语言、能勾选、有默认**，让立法回到"定标准"而不是"填表格"。
