# 标注结果回写设计：从软提示到确定性覆盖

> 目标：把人工标注的结果**确定性地回写到知识库内容**（如某个 wiki 页面），保证「标注过的问题不再复发」，同时不破坏「构建产物随 run 不可变、新内容=新包」的核心契约。
>
> 本文只讲**设计原理与思路**，不含落地代码。它是 `architecture-v2-roadmap.md` 第 5 节「标注样例进入构建 prompt」的下一步：把软提示升级为硬保证。

---

## 1. 问题背景

### 1.1 现状回顾

在当前的标注闭环里，一次人工标注会发生这些事（`knowledgeService.annotateReviewTask`）：

1. 写入 `annotation_examples`：记录 `(context_snapshot, context_hash, correct_value, page_type, rule_id)`；
2. 可选写入 `rule_dismissals`：标记「此规则对此组件不适用」；
3. 把 `review_tasks` 置为 `resolved`，记录 `annotation_value`；
4. 发出 `annotation.created` 事件。

下一次构建时，`extractStage` 会通过 `loadAnnotationExamplesForPrompt` 取最近 12 条标注，经 `buildSpecGuidance` 拼进 LLM 的 system prompt，作为 few-shot 参考。

### 1.2 这套机制的两个本质局限

**局限一：标注只是「软提示」，不保证生效。**

`buildSpecGuidance` 把标注样例作为「建议」拼进提示词。这意味着：

- LLM 可以**忽略**它 —— 同类问题仍可能复发；
- 当模型是 `deterministic`（无外部 API key 时的兜底）时，根本不读提示词，标注**完全不起作用**；
- few-shot 的影响是**概率性、全局性**的（最近 12 条混在一起影响所有页面），无法精确地说「这一页的这个字段，就必须是这个值」。

对「飞轮是否收敛」而言，软提示能减少复发率，但**给不出确定性保证**。用户真正想要的是：我标注过的页面，下次构建必须按我标注的来。

**局限二：内容不在数据库，且随 run 不可变。**

wiki 页面的 markdown 正文**不存在任何数据库列里**。`asset_components.storage_uri` 只存一个相对路径，真正的内容落在磁盘：

```
{dataDir}/kb-build-runs/{runId}/{storage_uri}
```

而且每次构建 = 一个**新** `asset_package` + 一批**新** `asset_components` 行，指向**新写的**磁盘文件。已有 run 的产物不会被就地修改（唯一例外是 publish 时 `releaseService` 回填 `quality` 列）。这是项目刻意维护的不可变契约（见 `CLAUDE.md`：`source_blobs`/`releases` 内容快照不可变，重导入走新版本）。

## 2. 设计张力

把上面两点放一起，就是这次设计要解决的核心矛盾：

> **用户要「确定性地把标注回写进 wiki 页面」；但页面内容由 LLM 每次构建重新生成，且随 run 不可变、不存在数据库里。**

由此可以排除两个直觉但错误的方案：

| 直觉方案 | 为什么不行 |
| --- | --- |
| **直接 patch 磁盘上的 `.md` 文件** | (1) 破坏「run 产物不可变」契约；(2) **下一次构建会用新生成的内容覆盖它**，标注白做；(3) 不经过包/组件/发布体系，无法审计、无法回滚、不进 OKF。 |
| **只靠加大 few-shot 权重 / 多塞样例** | 仍是概率性，deterministic 模型下完全失效，给不出「必然生效」的保证。 |

正确的思路要同时满足三条：**确定性生效** + **抗下次重建** + **不破坏不可变契约**。

---

## 3. 核心思路：把标注做成「extract 阶段的确定性覆盖（overlay）」

### 3.1 一句话原理

> 不要去改「构建产生的结果文件」，而要去改「产生结果的规则」。

具体说：在 extract 阶段，当 LLM（或确定性兜底）产出一个 `ExtractedPage`（`{type, title, source, facts, entities, relationships, body}`）之后、在把它渲染成磁盘 markdown（`renderWikiMarkdown`）之前，**插入一个确定性的覆盖步骤**：按「源文件路径 + 规则」匹配到对应的人工标注，把标注里的正确值**强制写入** `ExtractedPage` 的相应字段/段落。

```
读源 md → extractPage()/解析frontmatter → 【确定性 overlay：套用人工标注】 → normalizeTableRefs → renderWikiMarkdown → 写盘
                  ↑ 模型/兜底产物（可能不准）        ↑ 人工标注在这里"一锤定音"        ↑ 产出新组件
```

标注从「提示词里的建议」变成「extract 输出上的硬补丁」。

### 3.2 为什么这个思路对（与现有架构的契合点）

| 诉求 | overlay-at-extract 如何满足 |
| --- | --- |
| **确定性生效** | overlay 是普通代码逻辑，不经过模型自由发挥；deterministic 模型下同样生效。 |
| **抗下次重建（最关键）** | overlay 是「每次 extract 都会重放的规则」，不是一次性磁盘改写。无论构建多少次，只要标注还在，结果就一直被修正。 |
| **不破坏不可变契约** | 它仍然走一次**正常构建**，产出**新包/新组件/新文件**。已有 run 产物分毫不动。新内容=新包，完全符合既有模型。 |
| **可复现 / 可缓存** | 把 overlay 指纹并入 extract cache key 后，相同输入恒得相同输出；改了标注则缓存自然失效、重算。 |
| **零成本接入飞轮** | 产出新包后照旧触发 `build.completed → revision draft → OKF 复用 → 可选自动发布`，发布可回滚、可审计。 |
| **可审计** | 标注行 + `knowledge_events` + 构建 run 的 flywheel 摘要，记录「本次套用了几条 override」。 |

一句话：**它把「确定性回写」表达成「一次可复现的构建」，从而天然继承了系统已有的不可变、版本链、OKF 复用、发布回滚、可观测能力**，而不是在体系外开一个会被覆盖、无法回滚的旁路。

---

## 4. 标注 → 页面字段的映射模型

人工标注的「正确值」需要被解释成对 `ExtractedPage` 的一个**结构化局部补丁**。设计上区分覆盖粒度，保持强类型（不出现 `any`），并用 zod 校验：

| 覆盖目标 | 含义 | 典型对应规则 |
| --- | --- | --- |
| `setFacts` | 设/改某些 `facts[key] = value` | `wiki.required_fact`（必填事实缺失/写错） |
| `removeFacts` | 删除某些事实键 | 误抽取的字段 |
| `setTitle` | 覆盖页面标题 | 标题抽错 |
| `setType` | 覆盖页面类型（**须校验在 `specs.manifest.pageTypes` 内**，否则 extract 会把整页当 unknown 丢弃） | 页面类型判错 |
| `replaceSection` | 替换正文里某个 `## 标题` 段落的内容 | 局部正文需要人工定稿 |
| `replaceBody` | 整段正文替换 | 正文需要整体重写 |

**应用顺序固定**：type → title → facts → body/section，保证确定性。

**关键时序**：overlay 必须在 `normalizeTableRefs` **之前**应用 —— 这样人工改过的正文/事实，仍会经过表别名规范化、Data Dependencies 段对齐等既有后处理（复用 `alignDataDependenciesSection` 的段落扫描技巧来做 `replaceSection`），保持与普通页面一致的产物形态。

---

## 5. 软提示 vs 确定性覆盖：并存而非替换

不是用 overlay 取代 few-shot，而是**两层并存**：

- **`apply_mode = 'hint'`（默认，即现状）**：标注作为 few-shot 注入 prompt，影响是概率性、全局性的。适合"风格引导"类、不要求逐页精确的标注。
- **`apply_mode = 'override'`（新增）**：标注作为 extract 后的硬补丁，确定性、定点生效。适合"这一页这个字段必须是这个值"的定稿型标注。

这样既保留了软提示帮助模型整体收敛的价值，又为关键页面提供「一锤定音」的硬保证。存储上**复用 `annotation_examples` 表**，只加一个 `apply_mode` 列区分（不新建表），override 行同时也能继续作软提示用。

---

## 6. 触发与回流：标注后如何变成新版本

确定性 overlay 解决了「构建时如何生效」，还需要回答「标注完如何自动触发一次构建并发布」。这一步**完全复用现有飞轮**，不另起炉灶：

```
人工标注 (apply_mode=override)
        │  annotateReviewTask: 写 example(apply_mode=override) → 发事件
        ▼
annotation.writeback_requested  ← 新增事件类型
        │  新增监听器: 解析该组件的源文件路径 → 发起 scoped rebuild(only=该源, deterministic)
        ▼
scoped 构建 (extract 阶段套用 override) → 产出新包/新组件/新 wiki 正文
        │
        ▼
build.completed  ← 既有事件
        │  既有 releaseAutomationService 监听
        ▼
proposeRevisionDraftFromBuild → revision 草案(基于当前发布) → OKF 仅重写变化的 md
        │
        ▼
可选 autoMode 自动发布 (无 removed、无 blocking、trust 未下降) → release.published
```

**为什么不直接复用 `startRebuildFromReviewTask`**：它的门禁只认 `rule_id === 'agent_feedback.rebuild_candidate'` 且任务状态为 open；而回写任务已是 resolved，过不了门。因此把"按源路径发起 scoped rebuild"的核心抽成一个通用函数，回写监听器和原负反馈链路都复用它，各自保留自己的门禁/去重。

**防回环**：触发只来自人工标注；`build.completed` 不会再发 `annotation.writeback_requested`；再加上既有的 rebuild run 去重，不会自激。

## 7. 数据与可观测

- **存储**：复用 `annotation_examples`，新增 `apply_mode TEXT DEFAULT 'hint'`（走 `ALTER ... IF NOT EXISTS` 幂等迁移）。override 的结构化补丁随标注一并落库，供构建时重建。
- **缓存正确性**：extract cache key 必须并入 override 指纹 —— 否则改了标注却命中旧缓存，回写不生效。
- **可观测**：构建 run 的 `config.flywheel` 增「本次套用 override 数」；`knowledge_events` 记 `annotation.writeback_requested`；前端运行卡片可展示，让用户看到「标注确实驱动了重建」。

---

## 8. 风险与权衡

| 风险 | 对策 |
| --- | --- |
| **匹配错页**（slug/源路径漂移导致补丁打到别的页） | 只按**稳定的逻辑源路径 / `page.source`** 匹配，绝不按每次构建分配的 slug；sourcePath 在标注时就从 `context_snapshot.sourceFile` 或组件 `legacy_path` 固化下来。 |
| **override 与质量门禁冲突** | overlay 在 quality 阶段之前生效，门禁评的是覆盖后的页面。若覆盖后仍违反必填规则，需用户另行勾选规则豁免 —— **override 不自动豁免规则**，二者职责分离。 |
| **缓存陈旧** | override 指纹并入 extract cache key（见 §7）。 |
| **无限重建** | 触发仅来自人工标注 + rebuild 去重（见 §6）。 |
| **正确值语义歧义**（历史标注多是候选选择，不是补丁） | 只有 `apply_mode='override'` 的行被当硬补丁；历史 `hint` 行行为完全不变，向后兼容。 |

## 10. 数据结构 / 接口契约（设计草案）

> 以下为设计意图的精确化，便于评审；实现时以 `types.ts` / `schemas.ts` / `db.ts` 现有风格为准。

### 10.1 覆盖补丁 `AnnotationOverride`

标注「正确值」被解释成对 `ExtractedPage` 的局部补丁。强类型、无 `any`，构建时用 zod 校验后才套用：

```ts
// 匹配键：定位到具体哪一页、哪条规则
interface AnnotationOverride {
  sourcePath: string;   // 稳定逻辑源路径，如 "gamedocs/systems/pvp.md"；绝不用 slug
  ruleId: string;       // 如 "wiki.required_fact"
  pageType: string;     // 如 "activity"

  // 覆盖动作（均为可选，按 type→title→facts→body 顺序套用）
  setType?: string;                 // 须 ∈ specs.manifest.pageTypes，否则 warn 跳过
  setTitle?: string;
  setFacts?: Record<string, string>;
  removeFacts?: string[];
  replaceSection?: { heading: string; markdown: string }; // 替换某个 "## heading" 段
  replaceBody?: string;             // 整段正文替换
}
```

**校验要点**：字段全部 `trim` 且非空；`setType` 不在 page types 内时记 warning 并跳过该字段（不抛错，避免整页被当 unknown 丢弃）；解析失败的标注行**降级为 hint**，绝不让坏数据中断构建。

### 10.2 `annotation_examples` 扩展（复用表，仅加一列）

```sql
-- db.ts:migrate() 末尾的 ALTER 块，幂等
ALTER TABLE annotation_examples
  ADD COLUMN IF NOT EXISTS apply_mode TEXT NOT NULL DEFAULT 'hint';
-- 可选：按覆盖检索加速
CREATE INDEX IF NOT EXISTS idx_annotation_examples_override
  ON annotation_examples (apply_mode, page_type, rule_id, created_at DESC);
```

- `apply_mode ∈ {'hint','override'}`，默认 `'hint'`（= 现状，向后兼容）。
- override 的结构化补丁随标注一并落库：存进 `correct_value`（或 `context_snapshot.override`），构建时由 loader 还原成 `AnnotationOverride`。
- `types.ts` 的 `AnnotationExample` 增 `applyMode: "hint" | "override"`；`db/mappers.ts` 行映射同步补字段。

### 10.3 标注入参 `reviewAnnotationSchema` 扩展

`POST /api/review/tasks/annotate` 的 zod 入参在现有基础上增量：

```ts
{
  // ...现有字段：taskId / selectedCandidateId / correctValue / note / dismissRule / dismissalReason ...
  applyMode: z.enum(["hint", "override"]).default("hint"),
  override:  annotationOverrideSchema.partial().optional(), // 缺省时由 correctValue 推导
}
```

`annotateReviewTask` 据此：INSERT 时写 `apply_mode`；当 `applyMode==='override'`，COMMIT 后追加发 `annotation.writeback_requested` 事件（保留原 `annotation.created`）。

### 10.4 新增事件类型

`eventService.ts` 的 `KnowledgeEventType` 增一项：

| 事件 | entityType / entityId | payload | 用途 |
| --- | --- | --- | --- |
| `annotation.writeback_requested` | `review_task` / taskId | `{ componentId, sourcePath, versionId }` | 触发该组件源文件的 scoped rebuild |

### 10.5 构建侧新增/改造的函数签名（草案）

| 位置 | 函数 | 作用 |
| --- | --- | --- |
| `kbBuilder/extractStage.ts` | `applyAnnotationOverrides(page, overrides, specs, warnings): void` | extract 后、`normalizeTableRefs` 前，就地套用补丁 |
| `kbBuilder/extractStage.ts` | `extractCacheKey(...)` 增 `overrides` 入参 + `overridesFingerprint()` | 缓存正确性 |
| `kbBuilderService.ts` | `loadAnnotationOverrides(): Promise<AnnotationOverride[]>` | 取 `apply_mode='override'` 行并解析 |
| `kbBuilderService.ts` | `startScopedRebuildForSource(sourcePath, versionId, requestedBy, traceId, rebuildTaskId?)` | 从 `startRebuildFromReviewTask` 抽出的通用 scoped rebuild |
| `services/annotationWritebackService.ts`（新） | `registerAnnotationWriteback(...)` | 监听 `annotation.writeback_requested` → 调上一行 |

### 11.2 与 few-shot（hint）路径的分叉

```
                         annotateReviewTask
                                │
              apply_mode ?──────┴───────────────────────────┐
                  │ 'hint'                                   │ 'override'
                  ▼                                          ▼
        仅写 annotation_examples                  写 example + 发 writeback_requested
                  │                                          │
                  │（不主动触发构建）                          ▼
                  ▼                                  立即 scoped rebuild
        下次任意构建时                                         │
        loadAnnotationExamplesForPrompt                       │
        → buildSpecGuidance 注入 prompt                       ▼
        → 概率性影响（模型可忽略,                    extract overlay 确定性生效
          deterministic 不读）                       → 必然写入,定点,可复现
```

**读图要点**：hint 是「被动、概率、全局」；override 是「主动触发、确定、定点」。两者共用一张表、一个标注入口，靠 `apply_mode` 分流。

## 12. 小结

核心判断只有一句：**要确定性地回写内容，就去改"产生内容的规则"，而不是改"已产生的内容文件"。**

把人工标注表达成 extract 阶段的确定性 overlay，让它在每次构建时重放、产出新的不可变组件、走既有发布飞轮 —— 这样既给了用户想要的「标注必然生效、不再复发」，又一寸都没破坏系统赖以成立的不可变与版本化契约。
