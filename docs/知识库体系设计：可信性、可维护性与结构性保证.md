# 知识库体系设计：可信性、可维护性与结构性保证

> 从源知识到 Agent 消费的全链路信任机制

## 核心定位

这是一个把零散资料炼成 Agent 可信知识源的治理平台——资料进来、生成资产、审核证据、发布版本、回收反馈，循环演进。

## 设计哲学

把知识库当作**有合约、有测试、有版本、有 PR** 的代码仓库来治理：

- 源不可变
- 产出可追溯
- 发布不可变
- 问题可定位
- 反馈可闭环

---

## 一、可信性保证：从源头到消费的证据链

### 1. 源知识阶段：不可变 + 版本化

```
gamedocs/*.docx gamedata/*.xlsx → Source → SourceVersion (content_hash)
```

**为什么可信**

- 原始资料一旦进入平台就被哈希锁定，后续所有知识资产都能追溯到具体 `source_version_id`
- 不是"把文档扔进去"，而是记录**文档类型、章节结构、表格语义、修订历史**

**结构性保证**

- 原始资料永远不可变，修改必须产生新版本
- 避免"改了源文档但知识库不同步"的防腐设计

### 2. 知识生成阶段：规范驱动 + 可重现

```
SourceVersion → Run (adapter + config) → Artifact
```

**为什么可信**

- 每个 artifact 都知道自己是**哪个 adapter、哪个 config、哪个 source version、哪次 run** 生成的
- 相同 `(source, adapter, config)` 三元组理论上产出相同 artifact，这是调试和回归测试的基础

**结构性保证**

- 不是自由文本生成，而是按 `wiki_spec` 约束输出
- 每种知识资产（系统规则、表结构、数值公式、活动模板）都有**显式 Schema**
- LLM 必须填槽，不能自由发挥

### 3. 证据关联阶段：结论必带来源

```
Artifact.source_refs → 原始资料片段定位
```

**为什么可信**

- 每条知识结论（Wiki 段落、实体关系、表字段定义）都挂 `source_refs`
- 审核者能回查原文

**结构性保证**

- 证据不是"引用了某个文档"，而是**定位到页码/段落/sheet/cell**
- 未来可升级为 `EvidenceSpan`（原文摘录 + hash）
- 推断的、冲突的、来源不明的知识打 `low_confidence / inferred / conflict` 标签

## 二、可维护性保证：质量门禁 + 反馈闭环

### 4. 审核阶段：问题即任务

```
Artifact → Quality Gate → Findings (blocking / warning)
```

**为什么可维护**

- 质量报告生成**可操作的待修任务清单**
- 缺证据、引用断裂、孤儿页面、冲突实体，每个问题都标明"影响对象、建议动作、是否阻断发布"

**结构性保证**

- 质量门禁是**合约驱动**的
- 每个 artifact 类型有明确质量要求（如 `wiki_page` 必须有 `source_refs`，`table_schema` 的 FK 必须指向存在的表）
- 阻断级问题不解决就发布不了，避免知识库逐渐腐化

### 5. 发布阶段：不可变快照 + 回滚能力

```
Artifacts (approved) → Release (manifest_hash) → published
```

**为什么可维护**

- 发布版本是**不可变快照**
- Agent 永远读取确定版本，不会因为"后台在改"而返回半成品

**结构性保证**

- `release_manifest` 列出所有 artifact_id、它们的 hash、质量门禁结果、前序版本
- 这是**语义化版本控制**，而不是时间戳覆盖
- 可回滚：新版本有问题时，回退 `current_release_id` 指向即可

### 6. Agent 消费阶段：只读发布 + 审计追踪

```
MCP query → published Release → Artifact → source_refs
```

**为什么可维护**

- Agent 不能读草稿，保证消费的知识一定通过了质量门禁
- MCP 返回不只是答案，还附带**来自哪个 release、哪个 artifact、证据覆盖度、质量 flags**

**结构性保证**

- Agent 开发者能判断"这条知识靠不靠谱"
- 每次 Agent 查询都记录 `MCP Audit`（查了什么、命中了什么、用了哪个版本）
- 这是下一轮优化的输入

### 7. 反馈闭环：未命中 → 维护任务

```
MCP Audit (misses / low_quality_hits) → Feedback → 修订任务
```

**为什么可维护**

- Agent 反馈不是"用户抱怨"，而是**结构化失败记录**
- 未命中主题、低质量页面被频繁使用、重复查询同一关系——自动转成"补 topic_index / 补证据 / 强化图谱"任务

**错误本机制**（KC-ERRORBOOK-001）

- 重复失败必须进错误本
- 错误本不是日志，而是**持久纠错源**
- 下次重构或发布前必须处理

**飞轮驱动**

- 反馈不是旁路，而是知识进化的**主驱动力**
- Agent 用得越多、暴露问题越多、知识库越精准

## 三、结构性保证：从扁平文本到语义网络

### 8. 多层索引：不只是全文搜索

```
Wiki 页面 + Topic Index + Search Index + Table Ref Index + Entity Page Index
```

**为什么结构性强**

- 不是"把 Markdown 扔进向量库"，而是构建**多视角导航**
- 按主题找页面、按表名找关联 Wiki、按实体找解释页、按关键词搜索段落

**Index 是一等资产**

- 每种 index 都是可登记、可审核、可发布的 artifact
- 而不是临时派生物
- 保证了"目录烂了能修"，而不是"重新跑一遍"

### 9. 知识图谱：实体关系显式化

```
Graph: nodes (entities) + edges (relations) + indexes (adjacency)
```

**为什么结构性强**

- 不是自由文本推理，而是显式声明：
  - **实体类型**：system / table / resource / attribute
  - **关系类型**：depends_on / unlocks / configured_in
  - **关系方向和权重**

**可审核的图谱**

- 每个节点、每条边都带 `source_refs` 和置信度标签
- 孤立节点、悬空边、重复实体、低置信度关系都会被质量报告标出

**图谱 ≠ 可视化**

- `graph.json` 是**可查询的数据结构**
- Agent 能做"装备系统依赖哪些表""哪些活动产出金币"这种关系查询
- 而不只是人看一眼拓扑图

### 10. 表结构知识：从"配置表"到"一等资产"

```
Table Schema + Field Dictionary + FK Relations + Validation Report
```

**为什么结构性强**

- 不是"读个 xlsx"，而是把**表名、sheet、字段、类型、枚举、FK、单位、取值范围**全部登记为可审核资产

**表与 Wiki 双向引用**

- `table_ref_index` 记录"Wiki 页面引用了哪些表"、"表在哪些页面被解释"
- 这是游戏知识库的**核心结构骨架**

**字段级证据**

- 字段定义来自哪个文档、哪个章节、是推断的还是明确声明的
- 让"配表和文档对不上"的问题能被定位

## 总结：三层防护网

| 维度 | 源知识 | 知识资产 | Agent 消费 | 反馈修订 |
|------|--------|----------|------------|----------|
| **可信性** | 不可变版本 + hash | 规范驱动生成 + 证据挂钩 | 只读发布 + 审计追踪 | 结构化失败记录 |
| **可维护性** | 修改必出新版 | 质量门禁 + 问题即任务 | 不可变发布 + 回滚能力 | 错误本 + 飞轮驱动 |
| **结构性** | 文档类型 + 章节语义 | 多层索引 + 图谱 + 表结构 | 关系查询 + 多视角检索 | 未命中 → 补索引 |

## 知识进化飞轮

```
资料进入
  → 生成知识资产包
  → 审核证据和结构
  → 质量门禁
  → 发布给 Agent
  → 观察 Agent 使用反馈
  → 修订资料、索引、Wiki、图谱、表结构
  → 下一版发布
```

每一环的设计保证：

| 环节 | 普通用户看到 | 治理保证 |
|------|--------------|----------|
| 资料进入 | 上传资料、导入旧知识库 | 不可变版本、content hash、修订历史 |
| 生成资产 | 生成知识资产包 | 规范驱动、可重现、adapter + config 可追溯 |
| 审核 | 待补证据、待修冲突 | 质量门禁、问题即任务、阻断级检查 |
| 发布 | 当前 Agent 使用版本 | 不可变快照、manifest hash、回滚能力 |
| 使用反馈 | 未命中、低质量命中 | MCP audit、结构化失败记录、错误本 |
| 修订 | 重新生成、重新审核 | 版本演进、supersedes 关系、持久纠错源 |

---

## 最终目标

不是"生成一堆 Markdown"，而是**构建一个可被 Agent 信任、可被人类审核、可被持续演进的知识资产网络**。

让知识库从"一次性产物"变成"持续进化的资产"。
