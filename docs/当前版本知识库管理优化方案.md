# 当前版本知识库管理优化方案

> 目标：在不丢失旧知识库资产价值、不削弱治理能力的前提下，把当前知识平台从“工程对象驱动”优化为“知识资产驱动”。管理员和主开发者仍能看到完整追溯、质量、发布、MCP 细节；普通参与者也能理解知识库如何被共同维护、审核、发布和持续进化。

## 1. 当前判断

当前版本已经形成了治理底座：

- 事实源可以导入、哈希、版本化。
- 处理运行可以记录。
- artifact 可以登记、审核状态推进、进入 release。
- release 可以通过质量门禁后发布，且已发布版本不可覆盖。
- MCP 查询默认读取 published release，并返回质量与 trace。
- Web 和 CLI 已经覆盖主要管理动作。

但当前体验仍偏“工程台账”：

- 用户需要理解 `source / run / artifact / release / constitution / contract / quality gate / MCP` 才能顺畅操作。
- `artifact` 被感知为“一个登记文件”，但真实知识资产往往是一组内容：index 目录、wiki 页面、知识图谱、表结构、证据、质量报告、发布视图。
- 旧 `kb-builder` 资产被导入后，容易被拆散成孤立 artifact，用户难以看到“这批旧资产原本是什么、现在变成了什么、还有哪些缺口”。
- 审核、发布、Agent 使用反馈没有被包装成一个清晰的知识进化飞轮。

优化方向不是降低平台严谨性，而是做一层更友好的知识管理外观：

```text
内部模型：source -> run -> artifact -> release -> MCP audit
用户模型：资料 -> 知识资产包 -> 审核 -> 发布给 Agent -> 反馈修订
```

## 2. 设计原则

### 2.1 保留资产，不做平面化迁移

旧知识库中的 `index`、`wiki`、`graph`、`schema`、`_meta` 等内容都应被保留为可追溯资产，而不是只抽成若干 Markdown 文件。平台应记录：

- 原始目录位置。
- 原始文件类型。
- 原始分类。
- 生成来源。
- 与新平台 artifact 的映射关系。
- 缺失、冲突、低置信度、待审核项。

这样管理员和主开发者可以继续用熟悉的知识库结构定位问题，同时普通用户看到的是更清晰的知识资产包。

### 2.2 术语分层：用户看业务词，系统保留工程词

用户界面和日常文档优先使用业务词：

| 用户词 | 内部词 | 说明 |
|---|---|---|
| 资料 | Source / SourceVersion | 原始设计文档、表格、Markdown、历史导出 |
| 知识资产包 | Artifact Group / Asset Package | 一批由资料生成或导入的知识资产集合 |
| 知识页 | wiki_page artifact | 可阅读、可引用的 Wiki 页面 |
| 目录索引 | index artifact | 页面目录、主题索引、搜索索引、导航结构 |
| 知识图谱 | graph artifact | 实体、关系、邻接、图谱快照 |
| 表结构 | table_schema artifact | 配表字段、类型、外键、校验规则 |
| 证据 | Evidence / source_refs | 知识结论对应的来源片段 |
| 审核 | review status | 草稿、待复核、已通过、驳回 |
| 发布版本 | Release | Agent 正式消费的不可变版本 |
| Agent 查询记录 | MCP Audit | Agent 读了什么、来自哪个版本 |

工程词不隐藏，但放在详情页、JSON 视图、审计视图和开发者工具里。

### 2.3 默认流程要短，高级能力要完整

普通参与者需要看到 5 步：

```text
导入资料 -> 生成/导入知识资产包 -> 审核问题 -> 发布给 Agent -> 根据反馈修订
```

管理员和主开发者可以展开高级信息：

```text
source_version_id / run_id / artifact_id / release_id / config_version_id / quality profile / MCP trace
```

### 2.4 Artifact 应从“单文件记录”升级为“资产包组件”

当前 `artifact` 仍然应该保留为底层不可变记录，但用户层应引入“知识资产包”概念。一个资产包可以包含多个 artifact，并保留它们之间的结构关系。

例如一次旧知识库导入后，应形成：

```text
知识资产包：legacy_2026_06_10
  资料来源：
    - gamedocs/*.docx
    - gamedata/*.xlsx
  目录索引：
    - wiki index
    - topic lookup
    - search index
  Wiki 页面：
    - systems/*.md
    - activities/*.md
    - combat/*.md
    - numerical/*.md
    - tables/*.md
  知识图谱：
    - graph nodes
    - graph edges
    - relation index
  表结构：
    - table schemas
    - field dictionary
    - FK / enum / value constraints
  质量与证据：
    - evidence coverage
    - missing source refs
    - conflict report
    - stale report
```

这样 artifact 仍然是平台的治理单位，但用户理解的是“这是一包可审核、可发布、可回滚的知识资产”。

## 3. Artifact 体系优化

### 3.1 Artifact 的用户层定义

建议把 artifact 在产品中解释为：

> 知识资产是由资料生成、人工整理或旧知识库导入的一份可追溯内容。它可以是一篇 Wiki、一个目录索引、一张知识图谱、一份表结构说明，也可以是一份质量报告。多个相关知识资产组成一个知识资产包。

这个定义比“产物登记记录”更容易理解，也更接近实际知识库资产。

### 3.2 Artifact 类型分组

建议把 `artifact_type` 分成几组，而不是让用户面对一串技术枚举。

| 分组 | artifact_type 示例 | 用户理解 |
|---|---|---|
| 阅读资产 | `wiki_page`, `standard_doc`, `module_doc` | 人直接阅读和审核的知识内容 |
| 导航资产 | `index`, `topic_index`, `page_tree`, `search_index` | 帮助定位知识的目录和索引 |
| 结构资产 | `table_schema`, `field_dictionary`, `entity_schema` | 表、字段、实体类型、约束 |
| 图谱资产 | `graph`, `graph_nodes`, `graph_edges`, `relations` | 实体关系网络 |
| 证据资产 | `evidence_pack`, `source_map`, `citation_map` | 来源、引用、证据片段 |
| 质量资产 | `quality_report`, `coverage_report`, `conflict_report` | 质量门禁和问题列表 |
| 发布资产 | `release_manifest`, `mcp_read_model` | Agent 消费视图 |

底层可以继续使用现有 `artifact_type` 字段；前端和文档提供分组视图即可。

### 3.3 知识资产包

新增概念：`asset_package`，用于把同一批知识资产组织起来。

推荐字段：

```json
{
  "package_id": "pkg_legacy_2026_06_10",
  "name": "旧知识库导入 2026-06-10",
  "kind": "legacy_import",
  "status": "draft",
  "source_version_ids": ["srcv_xxx"],
  "artifact_ids": ["art_index_xxx", "art_wiki_xxx", "art_graph_xxx"],
  "legacy_paths": ["wiki/systems", "wiki/_meta", "graph"],
  "created_by_run_id": "run_xxx",
  "quality_summary": {
    "overall_score": 0.82,
    "blocking_count": 3,
    "warning_count": 12
  }
}
```

这不是替代 artifact，而是给 artifact 增加集合边界。好处：

- 管理员可以按“一次导入 / 一次生成 / 一个模块”管理知识。
- 主开发者可以看到每个包包含哪些底层 artifact。
- 普通用户不需要先理解 `artifact_id`，只需要知道“这个知识包有哪些问题，能不能发布”。

### 3.4 Index 作为一等资产

旧知识库通常有多种 index：

- 页面目录。
- 主题到页面的映射。
- 关键词索引。
- 表与 Wiki 页面的引用关系。
- 图谱节点到页面的反查。
- Agent 查询时使用的搜索索引。

这些不能只作为临时派生产物。建议把 index 拆成可登记 artifact：

| index 类型 | 用途 | 审核关注 |
|---|---|---|
| `page_tree` | 页面层级和分类导航 | 是否漏页、分类是否合理 |
| `topic_index` | 主题词到页面/实体/表的映射 | 同义词、别名、歧义 |
| `search_index` | Agent 检索用 | 覆盖率、失效引用 |
| `table_ref_index` | Wiki 与配表关系 | 页面引用的表是否存在 |
| `entity_page_index` | 图谱实体对应页面 | 实体是否有解释页 |

用户界面中可以叫“目录索引”，详情里再显示具体类型。

### 3.5 Wiki 页面作为阅读资产

Wiki 页面仍然是最容易被人审核的知识资产。优化重点不是改变它，而是补齐上下文：

- 页面属于哪个知识资产包。
- 页面来自哪些资料版本。
- 页面由哪个处理运行生成。
- 页面符合哪个 wiki spec。
- 页面引用了哪些表、实体、关系。
- 页面有哪些证据缺口和低置信度段落。
- 页面是否进入某个 release。
- Agent 最近是否查询过它。

页面列表不应只显示 `artifact_id`，而应显示：

```text
标题 | 分类 | 来源资料 | 审核状态 | 质量分 | 证据覆盖 | 最近发布版本 | Agent 使用次数
```

### 3.6 知识图谱作为结构资产

知识图谱不应只是一份 graph 文件。建议拆成三层：

```text
Graph Snapshot
  - nodes: 实体列表
  - edges: 关系列表
  - indexes: 邻接、反查、实体到页面映射
```

平台需要保留：

- 每个实体的来源和证据。
- 每条关系的来源和证据。
- low confidence / inferred / conflict 标记。
- 孤立节点、悬空边、重复实体报告。
- 图谱快照所属 release。

用户界面中可以叫“知识图谱”，但详情要允许主开发者下载或查看：

- 节点 JSON。
- 边 JSON。
- 邻接关系。
- 图谱质量报告。
- 图谱与 Wiki 页面的双向映射。

### 3.7 表结构作为高价值资产

配表知识对游戏知识库非常关键，应从“表读取工具”提升为平台资产。

建议 artifact 类型包括：

- `table_schema`：表名、sheet、字段、类型、必填、默认值。
- `field_dictionary`：字段含义、枚举、单位、取值范围。
- `table_relation`：FK、引用关系、表族关系。
- `table_sample`：安全样例，不一定存完整业务数据。
- `table_validation_report`：缺字段、类型错误、引用断裂。

用户看到的是“表结构资产”，主开发者可展开字段级详情。

### 3.8 证据资产单独可见

当前 artifact 有 `source_refs`，但用户很难理解证据覆盖情况。建议新增“证据视图”：

```text
知识结论 -> 来源资料 -> 定位方式 -> 原文片段 -> 审核状态
```

短期即使还没有完整 `EvidenceSpan` 模型，也可以先做过渡：

- `source_refs` 标明来源版本。
- 页面 frontmatter 或正文保留 `evidence_refs`。
- 质量报告统计“无证据知识数”。
- Web 中按页面展示“证据完整 / 缺证据 / 证据待确认”。

### 3.9 质量资产让问题可修

质量报告不应只是一份 JSON。它应被用户理解为“待修问题列表”。

每个质量问题至少需要：

```text
问题级别 | 影响对象 | 人能看懂的描述 | 建议动作 | 相关资料 | 相关知识资产 | 是否阻断发布
```

示例：

```text
阻断：装备系统页面缺少证据引用
影响：wiki_page / 装备系统
建议：补充 source_refs 或 evidence_refs 后重新审核
```

这样质量门禁从“工程检查”变成“知识维护任务”。

## 4. 用户体验优化

### 4.1 Web 导航重命名

当前导航建议从工程模块改成飞轮流程：

| 当前入口 | 建议入口 | 说明 |
|---|---|---|
| 概览 | 首页 | 显示知识库健康度、最新发布、待处理问题 |
| 事实源 | 资料库 | 上传、导入、版本、原始资料 |
| 处理运行 | 生成记录 | 默认折叠，高级用户查看 |
| 知识产物 | 知识资产 | 按资产包、Wiki、Index、Graph、表结构分组 |
| Releases | 发布 | 候选版本、发布版本、回滚指向 |
| Doctor | 质量检查 | 门禁结果 + 待修问题 |
| Legacy 导入 | 旧知识库导入 | 向导化，导入后生成资产包 |
| MCP 控制台 | Agent 查询 | 工具调试、查询审计、Agent 使用反馈 |

### 4.2 首页展示飞轮，而不是模块清单

首页建议展示：

```text
资料库：xx 份资料，x 份最近更新
知识资产：xx 个资产包，x 个待审核
质量问题：x 个阻断，x 个警告
发布版本：当前 rel_xxx，包含 x 个知识资产
Agent 使用：最近 x 次查询，命中 x 个知识页
```

并提供 3 个主按钮：

- 导入资料。
- 审核知识。
- 发布给 Agent。

### 4.3 旧知识库导入向导

旧资产导入应向导化：

1. 选择旧 `kb-builder data` 目录。
2. 扫描并预览：发现多少资料、Wiki 页面、Index、Graph、表结构。
3. 映射为知识资产包。
4. 显示缺失来源、孤儿页面、无证据内容。
5. 一键导入为 draft。
6. 进入审核中心。

导入结果不应只返回 `sources_imported / artifacts_registered`，还应有用户视角摘要：

```text
已导入：
- 42 份资料
- 126 篇 Wiki 页面
- 5 个目录索引
- 1 份知识图谱
- 37 张表结构

需要处理：
- 9 篇页面缺来源
- 14 条关系缺证据
- 3 个 Index 指向不存在页面
```

### 4.4 审核中心以问题驱动

审核中心建议有三个视图：

- 按资产包：适合管理员看一批知识是否能发布。
- 按问题：适合普通用户领任务修。
- 按类型：Wiki / Index / Graph / 表结构。

审核动作保持简单：

```text
通过 | 驳回 | 需要补证据 | 标记冲突 | 标记已过期
```

内部仍映射为 `draft / reviewed / approved / rejected / stale / conflict` 等状态。

### 4.5 发布页只回答三个问题

发布页不要让用户先理解 release manifest。它应先回答：

1. 这次会发布哪些知识资产？
2. 还有哪些阻断问题？
3. 发布后 Agent 会读到什么？

高级展开再显示：

- `release_id`
- `artifact_ids`
- `manifest_hash`
- `quality_gate_result`
- `previous_release_id`

### 4.6 Agent 查询页变成反馈入口

MCP 控制台不只是调试工具，也应成为知识进化飞轮的反馈入口。

展示：

- Agent 查了什么问题。
- 命中了哪些知识页、图谱节点、表结构。
- 使用的是哪个 release。
- 返回了哪些质量 flags。
- 哪些查询没有命中。
- 哪些低质量知识被频繁使用。

这些反馈可以转成维护任务：

```text
未命中主题 -> 补 topic_index
低质量页面被频繁使用 -> 优先补证据
Agent 反复查同一关系 -> 检查图谱是否需要强化
```

## 5. CLI 优化

CLI 可以保留当前 `kb` 统一入口，但新增更贴近飞轮的复合命令。

### 5.1 推荐命令

```bash
kb status
kb import legacy D:/projects/knowledge/data
kb assets list
kb assets show pkg_legacy_2026_06_10
kb review list --blocking
kb publish --version 2026.06.10.001
kb agent audit --limit 20
```

底层命令继续保留：

```bash
kb source
kb run
kb artifact
kb release
kb mcp
```

但日常文档优先讲复合命令。

### 5.2 `kb status` 增强

当前 `kb status` 已经很好，建议增加资产包和质量摘要：

```text
data:           D:/...
current release: rel_2026_06_10_001

资料库         active=42 superseded=3
知识资产包     total=6 draft=2 approved=3 published=1
知识资产       wiki=126 index=5 graph=1 table_schema=37
待修问题       blocking=3 warning=12
Agent 查询      today=84 misses=9 low_quality_hits=4
```

## 6. 数据模型补强建议

### 6.1 短期：在现有 artifact 上加字段

可以先不引入数据库，只在 artifact JSON 中增加可选字段：

```json
{
  "asset_package_id": "pkg_legacy_2026_06_10",
  "display_name": "装备系统",
  "asset_group": "wiki",
  "asset_kind": "system_rule",
  "legacy_path": "wiki/systems/equipment.md",
  "logical_path": "systems/equipment",
  "linked_artifact_ids": ["art_topic_index_xxx", "art_graph_xxx"],
  "review_summary": {
    "review_status": "approved",
    "reviewer": "admin",
    "reviewed_at": "2026-06-10T00:00:00Z"
  }
}
```

这些字段不会破坏现有契约，却能显著提升 UI 组织能力。

### 6.2 中期：新增资产包记录

新增：

```text
data/packages/<package_id>.json
```

它聚合 sources、runs、artifacts、quality summary 和 legacy mapping。

### 6.3 中期：新增证据记录

新增：

```text
data/evidence/<evidence_id>.json
```

用于承载：

- `source_version_id`
- 定位方式：页码、段落、sheet/cell、标题、行号。
- 引文摘要。
- hash。
- 审核状态。

### 6.4 中期：新增反馈记录

新增：

```text
data/feedback/<feedback_id>.json
```

来源可以是：

- MCP 未命中。
- Agent 低质量命中。
- 人工评论。
- 发布后问题。
- 重复失败。

反馈应能关联到 package、artifact、release、source。

## 7. 知识库进化飞轮

最终希望所有用户都能理解并参与这个飞轮：

```text
资料进入
  -> 生成知识资产包
  -> 审核证据和结构
  -> 质量门禁
  -> 发布给 Agent
  -> 观察 Agent 使用反馈
  -> 修订资料、索引、Wiki、图谱、表结构
  -> 下一版发布
```

每一环的用户语言：

| 环节 | 普通用户看到 | 管理员/主开发者可展开 |
|---|---|---|
| 资料进入 | 上传资料、导入旧知识库 | source hash、source_version_id |
| 生成资产 | 生成知识资产包 | run_id、adapter、config_version_id |
| 审核 | 待补证据、待修冲突 | artifact status、quality profile |
| 发布 | 当前 Agent 使用版本 | release manifest、manifest_hash |
| 使用反馈 | 未命中、低质量命中 | MCP audit、trace、quality flags |
| 修订 | 重新生成、重新审核 | supersedes、stale、previous_release_id |

这个飞轮比“跑一串命令”更容易让多人协作。

## 8. 推荐实施顺序

### 第一阶段：不改核心，只改表达

- 更新文档和 Web 文案，把 `事实源/知识产物/Releases/MCP` 映射成 `资料库/知识资产/发布/Agent 查询`。
- 修订 `docs/重构进展对照.md`，移除过期状态。
- 在操作手册开头加入“知识进化飞轮”。
- 在 artifact 文档中补充 Index、Wiki、Graph、表结构等资产类型说明。

### 第二阶段：资产包视图

- 在 artifact JSON 中增加可选 `asset_package_id / display_name / asset_group / legacy_path`。
- Legacy 导入时自动生成 package summary。
- Web 知识资产页按资产包和资产类型分组。
- `kb status` 增加资产包和待修问题摘要。

### 第三阶段：审核与质量任务化

- 把质量门禁 findings 转成“待修任务”视图。
- 按问题级别、资产包、资产类型筛选。
- 支持“通过 / 驳回 / 需要补证据 / 标记冲突”。

### 第四阶段：Evidence 与反馈闭环

- 增加 EvidenceSpan 记录。
- MCP audit 中增加命中 artifact、quality flags、misses 统计。
- Agent 查询页支持把未命中和低质量命中转成维护任务。

### 第五阶段：标准 MCP 和 Agent 迁移

- 把当前 dispatcher 包成标准 MCP stdio server。
- 扩充 `kb_*` 工具：页面、索引、图谱、表结构、证据。
- `game-designer` 并行接入后逐步下线本地知识读取工具。

## 9. 架构边界

这些优化不应破坏现有防腐规则：

- Web 只做用户体验和 API 适配，不直接改写核心数据。
- MCP 只读 published release，不读取草稿资产。
- 资产包是 artifact 的组织视图，不替代 source、artifact、release 的核心契约。
- 旧知识库导入保留 legacy path 和原始结构，但不能让旧目录重新成为事实源。
- 发布版本仍然不可变，修复必须创建新版本。
- Agent 反馈进入 feedback/task，不直接修改知识资产。

## 10. 验收标准

优化完成后，应该满足：

- 普通用户能用 5 分钟理解：资料、知识资产、审核、发布、反馈。
- 管理员能一眼看到：当前知识库健康度、待修问题、当前发布版本。
- 主开发者能追溯：source_version_id、run_id、artifact_id、release_id、MCP audit。
- 旧知识库资产没有丢失：index、wiki、graph、schema、meta 都能看到来源和映射。
- Artifact 不再被理解为孤立文件，而是知识资产包中的可审核组件。
- Agent 查询反馈能回流为知识维护任务，形成持续进化飞轮。
