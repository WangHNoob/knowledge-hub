# 知识库体系重构实施方案

> 目标：在保留现有 `kb-builder` 资产价值的基础上，将知识库从离线脚本产物升级为可管理、可审核、可发布、可评估，并通过统一 Knowledge MCP 被 Agent 消费的知识平台。

## 1. 结论

当前 `docs/知识库体系重构计划.md` 的方向正确，但需要从“能力清单”收束成一套可落地架构：

1. Web、CLI、MCP 只作为入口，不能拥有知识业务逻辑。
2. 事实源、标准知识、审核记录、发布版本必须成为平台核心数据。
3. Wiki、Index、Graph、Schema 是可重建派生产物，不再是唯一事实源。
4. Agent 只能消费已发布版本，查询结果必须返回来源、版本、质量状态和证据。
5. `game-designer` 已经具备 MCP Client 能力，Knowledge MCP 应优先以独立 MCP Server 接入，而不是继续让 Agent 读取知识库目录。

建议采用“平台化包裹 + 分层替换”的路线，不建议一次性推倒重写现有流水线。

## 2. 当前系统观察

### 2.1 kb-builder 现状

当前仓库仍是离线流水线：

```text
convert -> extract -> tables -> graph -> viz
```

主要代码位置：

| 模块 | 当前职责 | 新体系定位 |
|---|---|---|
| `scripts/doc_reader.py` | docx/xlsx/md 解析，带缓存 | 文档解析适配器 |
| `scripts/extract/wiki_extractor.py` | LLM 生成 Wiki 页面、实体、关系 | 知识抽取适配器 |
| `scripts/extract/table_analyzer.py` | 扫描配置表字段、表族、FK | 表知识能力适配器 |
| `scripts/graph/graph_builder.py` | 合并 `_meta` 生成 graph/index | 派生产物构建适配器 |
| `templates/wiki_specs/` | Wiki 页面规范 | 项目配置初始模板 |
| `data/wiki/` | 最终生成物 | 发布版本或导出视图 |

需要特别处理的风险：

- `doc_reader.py` 目前可能原地修复 docx。新体系中事实源不可变，修复必须发生在导入副本或解析工作区。
- `_meta/*.json` 目前是半结构化加工记录，但还不足以承载审核、版本、证据、质量门禁。
- `data/wiki/` 当前会被流水线覆盖，新体系中应区分 `draft`、`reviewed`、`published`。

### 2.2 game-designer 工具现状

`D:/design-agent/game-designer` 当前知识工具集中在 `com.gametoolbox.tool.common`。

现有只读知识工具：

| 当前工具 | 当前能力 | MCP 迁移建议 |
|---|---|---|
| `wiki_read` | 读取 Wiki 页面 | 迁移到 `kb_get_page` |
| `wiki_read_section` | 读取 Wiki 章节 | 迁移到 `kb_get_section` |
| `wiki_list` | 按分类列出页面 | 迁移到 `kb_list_pages` |
| `wiki_lookup` | 主题到页面映射 | 迁移到 `kb_resolve_topic` |
| `wiki_relations` | 查询主题关系 | 迁移到 `kb_get_relations` |
| `wiki_table_refs` | 查询页面涉及的表 | 迁移到 `kb_get_page_tables` |
| `kg_query_node` | 图谱节点详情 | 迁移到 `kb_get_entity` |
| `kg_query_neighbors` | 图谱邻接关系 | 迁移到 `kb_get_neighbors` |
| `kg_list_nodes` | 按类型列节点 | 迁移到 `kb_list_entities` |
| `grep_search` | 本地 Markdown grep | 迁移到 `kb_search` |
| `table_list` | 配表发现 | 迁移到 `kb_list_tables` |
| `table_read` | 读取配表 | 迁移到 `kb_get_table` |
| `table_query` | 查询配表数据 | 迁移到 `kb_query_table` |
| `table_validate` | 校验配表 | 迁移到 `kb_validate_table` |
| `table_check_cell` | 检查字段取值 | 迁移到 `kb_check_table_value` |

不建议迁入 Knowledge MCP 的工具：

| 当前工具 | 原因 |
|---|---|
| `table_copy_to_workspace` | 属于 Agent 工作区编辑，不是知识库只读消费 |
| `table_create` | 会创建业务产物，应继续归工作区或配置表编辑流程管理 |
| `table_write` | 写操作不能进入默认知识查询 MCP |
| `table_write_headers` | 同上 |
| `table_add_sheet` | 同上 |
| `field_lookup`、`field_conflicts`、`field_report` | 这是 Agent 会话内字段注册表，生命周期属于 `game-designer` session，不属于全局知识库 |
| `workspaceTool`、`documentTool`、`blackboardTool` | 属于 Agent 工作过程，不属于知识平台事实源 |

`game-designer` 已经有 `McpConfig`，支持 `stdio`、`sse`、`http` 三种传输；`AgentRegistry` 会把配置中的 MCP tools 注册进 Toolkit，并支持 `enableTools`、`disableTools`。因此知识库侧可以先提供一个独立 `knowledge-mcp`，Agent 端通过配置接入。

迁移注意点：

- 初期不要让 MCP 暴露和本地工具同名的 `wiki_read`、`kg_query_node` 等，避免 Toolkit 工具名冲突。
- 第一阶段 MCP 使用 `kb_*` 新工具名并行接入。
- 验证稳定后，在 `game-designer` 中禁用本地知识工具，或者保留旧工具为调用 MCP 的兼容壳。

## 3. 目标架构

采用端口-适配器架构：

```text
Web UI / CLI / Knowledge MCP
        |
        v
Application Services
        |
        v
Domain Core
        |
        v
Ports
        |
        v
Adapters: filesystem, SQLite/Postgres, parser, LLM, search, graph, MCP protocol
```

依赖方向：

- `domain` 不依赖 Web、MCP、FastAPI、AgentScope、LLM SDK。
- `application` 编排用例，依赖领域模型和端口。
- `adapters` 实现文件系统、数据库、LLM、解析器、搜索索引。
- `mcp` 只做协议适配和权限边界。
- `web` 只做页面和 API 适配。

建议新目录：

```text
kb_platform/
  domain/
    models.py
    states.py
    policies.py
  application/
    source_service.py
    processing_service.py
    artifact_service.py
    quality_service.py
    release_service.py
    query_service.py
  ports/
    repositories.py
    parser.py
    extractor.py
    artifact_builder.py
    search_index.py
    clock.py
  adapters/
    storage_sqlite.py
    storage_filesystem.py

    markitdown_parser.py

```

## 4. 核心领域模型

首期至少需要以下模型。

### 4.1 Project

表示一个知识库项目。

关键字段：

- `project_id`
- `name`
- `default_release_id`
- `created_at`
- `updated_at`

### 4.2 ProjectConfigVersion

项目组定义的知识标准版本。

关键字段：

- `config_version_id`
- `project_id`
- `version`
- `document_classes`
- `module_specs`
- `table_rules`
- `entity_types`
- `relation_types`
- `wiki_templates`
- `index_rules`
- `graph_rules`
- `quality_rules`
- `status`

所有处理结果都必须记录 `config_version_id`。

### 4.3 SourceDocument 与 SourceVersion

事实源和具体版本分离。

关键字段：

- `source_id`
- `source_version_id`
- `project_id`
- `source_type`
- `original_filename`
- `storage_uri`
- `content_hash`
- `imported_at`
- `imported_by`
- `status`

事实源原始文件不可变。任何修复、解析、清洗都写入处理工作区或派生产物。

### 4.4 ProcessingJob 与 ProcessingRun

用于追踪解析、抽取、生成、校验任务。

关键字段：

- `job_id`
- `job_type`
- `input_refs`
- `config_version_id`
- `status`
- `started_at`
- `finished_at`
- `error`
- `metrics`
- `output_artifact_ids`

### 4.5 KnowledgeItem

标准知识的最小单位，可以是实体、关系、事实、表字段、模块段落。

关键字段：

- `knowledge_id`
- `kind`: `entity`、`relationship`、`fact`、`table_schema`、`section`
- `name`
- `content`
- `status`: `confirmed`、`missing`、`ambiguous`、`conflict`、`inferred`、`not_applicable`
- `review_status`: `draft`、`approved`、`rejected`
- `source_refs`
- `evidence_refs`
- `quality_profile`
- `created_by_run_id`

### 4.6 EvidenceSpan

证据必须从一开始建模，不能后补。

关键字段：

- `evidence_id`
- `source_version_id`
- `locator_type`: `page`、`paragraph`、`sheet_cell`、`markdown_heading`、`line_range`
- `locator`
- `quote`
- `hash`

### 4.7 Artifact

所有派生产物统一管理。

关键字段：

- `artifact_id`
- `artifact_type`: `standard_doc`、`wiki_page`、`table_schema`、`index`、`graph`、`quality_report`
- `project_id`
- `status`: `draft`、`reviewed`、`published`、`stale`
- `version`
- `storage_uri`
- `source_knowledge_ids`
- `created_by_run_id`
- `config_version_id`
- `quality_profile`

### 4.8 Release

Agent 默认只查询 Release。

关键字段：

- `release_id`
- `project_id`
- `version`
- `artifact_ids`
- `quality_gate_result`
- `published_at`
- `published_by`
- `previous_release_id`

## 5. 知识处理流程

新流程：

```text
事实源导入
 -> 解析草稿
 -> 文档分类与完整性评估
 -> 标准知识抽取
 -> 人工审核
 -> 派生产物生成
 -> 一致性校验
 -> 发布 Release
 -> Agent 通过 MCP 查询 Release
```

关键约束：

1. 加工结果先进入草稿区。
2. 审核通过后才能进入待发布集合。
3. 发布前必须跑质量门禁。
4. 发布版本不可变。
5. 上游事实源或配置变化后，下游产物标记 `stale`，不直接覆盖。

## 6. 质量与可信度模型

复用已有 K-Cred 设计，将它作为平台质量模型的基础。

首期质量维度：

| 维度 | 评估对象 | 说明 |
|---|---|---|
| `source_authority` | 事实源 | 来源权威等级 |
| `extraction_confidence` | 实体、关系、事实 | LLM 抽取置信度 |
| `evidence_coverage` | 标准知识、Wiki | 每条结论是否有证据 |
| `structure_score` | Wiki、标准文档 | 必需章节覆盖率 |
| `table_schema_score` | 表 Schema | 表头识别、字段类型、FK 可信度 |
| `index_coverage` | Index | 页面覆盖率、失效链接 |
| `graph_integrity` | Graph | 悬空边、孤立节点、关系来源 |
| `review_score` | 全部知识项 | 人工审核状态 |
| `freshness` | Release | 是否基于最新事实源和配置 |

发布门禁建议：

```text
P0: 不能发布
- 存在无来源的已确认知识
- 存在 graph dangling edge
- Index 指向不存在页面
- Wiki 页面缺少 required frontmatter
- 待发布 artifact 引用 stale source/config

P1: 可以发布但必须提示
- 低置信度实体或关系进入图谱
- 文档完整性缺失
- 表字段识别置信度低
- 存在孤立节点

P2: 记录观察
- WikiLink 密度低
- 页面内容过短
- 单来源事实较多
```

所有 MCP 查询结果都应返回质量信息，而不是只返回正文。

## 7. Knowledge MCP 设计

### 7.1 服务形态

建议首期实现一个 `knowledge-mcp`，而不是拆成多个 MCP Server。

理由：

- 权限、发布版本、质量门禁、审计日志需要统一。
- `game-designer` 已经支持多个 MCP，但多个知识 MCP 会让工具发现和版本一致性变复杂。
- 一个 server 可以用工具命名空间表达能力边界。

传输方式：

1. 首期优先 `stdio`，便于 `game-designer` 本地启动。
2. 平台 Web/API 稳定后增加 `streamable HTTP`，便于远程 Agent 和多用户部署。

### 7.2 工具命名

首期使用 `kb_*`，避免与 `game-designer` 现有本地工具名冲突。

只读消费工具：

| MCP 工具 | 用途 | 替代旧工具 |
|---|---|---|
| `kb_search` | 语义或关键词检索已发布知识 | `grep_search` |
| `kb_resolve_topic` | 主题映射到页面、实体、表 | `wiki_lookup` |
| `kb_get_page` | 获取 Wiki 页面 | `wiki_read` |
| `kb_get_section` | 获取 Wiki 章节 | `wiki_read_section` |
| `kb_list_pages` | 按类型列页面 | `wiki_list` |
| `kb_get_page_tables` | 查询页面引用的表 | `wiki_table_refs` |
| `kb_get_entity` | 获取实体详情 | `kg_query_node` |
| `kb_get_neighbors` | 获取邻接关系 | `kg_query_neighbors` |
| `kb_list_entities` | 按类型列实体 | `kg_list_nodes` |
| `kb_get_relations` | 查询主题关系 | `wiki_relations` |
| `kb_list_tables` | 列出配置表 | `table_list` |
| `kb_get_table_schema` | 获取表 Schema | `table_read` 的 schema 部分 |
| `kb_query_table` | 查询表数据或样例 | `table_query` |
| `kb_validate_table` | 校验表结构 | `table_validate` |
| `kb_check_table_value` | 检查字段值是否合法 | `table_check_cell` |
| `kb_get_quality` | 查询知识、页面、表、图谱质量 | 新增 |
| `kb_get_evidence` | 查询来源证据 | 新增 |
| `kb_get_release` | 查询当前发布版本 | 新增 |

管理工具不建议在首期暴露给 Agent，后续可以做受权限保护的管理 MCP：

| 管理工具 | 用途 |
|---|---|
| `kb_register_source` | 登记事实源 |
| `kb_start_processing_job` | 启动处理任务 |
| `kb_get_job_status` | 查询任务状态 |
| `kb_review_artifact` | 审核产物 |
| `kb_publish_release` | 发布版本 |
| `kb_rollback_release` | 回滚版本 |

### 7.3 返回契约

所有只读工具返回统一 envelope：

```json
{
  "ok": true,
  "release": {
    "release_id": "rel_2026_06_10_001",
    "version": "2026.06.10.001"
  },
  "data": {},
  "source_refs": [
    {
      "source_id": "src_equipment",
      "source_version_id": "srcv_001",
      "title": "装备异化.docx",
      "evidence_ids": ["ev_001"]
    }
  ],
  "quality": {
    "overall_score": 0.86,
    "status": "approved",
    "flags": []
  },
  "trace": {
    "artifact_id": "art_wiki_equipment",
    "config_version_id": "cfg_001",
    "generated_by_run_id": "run_001"
  }

```

失败返回：

```json
{
  "ok": false,
  "error": {
    "code": "NOT_FOUND",
    "message": "未找到主题：装备异化",
    "retryable": false
  }

```

### 7.4 Agent 查询审计

Knowledge MCP 应记录：

- `session_id`
- `agent_role`
- `tool_name`
- `query`
- `release_id`
- `returned_artifact_ids`
- `returned_quality_flags`
- `latency_ms`

`game-designer` 已经使用 O11y 上下文和 Blackboard 缓存，MCP 返回中应保留 `trace`，方便后续引用审计和质量回放。

## 8. game-designer 迁移方案

### 8.1 第一阶段：并行接入

在 `application.yml` 中新增 Knowledge MCP 配置，示例：

```yaml
gametoolbox:
  mcp:
    enabled: true
    servers:
      - name: knowledge
        enabled: true
        transport: stdio
        command: uv
        args:
          - run
          - python
          - -m
          - kb_platform.mcp.server
        enable-tools:
          - kb_search
          - kb_resolve_topic
          - kb_get_page
          - kb_get_section
          - kb_get_entity
          - kb_get_neighbors
          - kb_list_tables
          - kb_get_table_schema
          - kb_query_table
          - kb_get_quality
          - kb_get_evidence
```

本阶段保留本地 `wiki_*`、`kg_*`、`table_*` 工具，Prompt 中逐步引导 Agent 优先使用 `kb_*`。

### 8.2 第二阶段：兼容壳

将 `WikiPageTool`、`KnowledgeGraphTool`、`ConfigTableQueryTool` 改为可选兼容壳：

- 如果 `knowledge-mcp` 可用，旧工具内部调用 MCP。
- 如果 MCP 不可用，短期回退本地文件。
- 回退路径记录告警，避免长期双实现。

### 8.3 第三阶段：下线本地知识读取

`createReadOnlyToolkit` 中移除：

- `grepSearchTool`
- `knowledgeGraphTool`
- `wikiPageTool`
- `configTableQueryTool`

保留：

- `workspaceTool`
- `documentTool`
- `blackboardTool`
- `fieldRegistryTool`
- `TavilySearchTool`

`createTableToolkit` 中只保留工作区写表工具，不再直接读取全局知识目录。

### 8.4 Prompt 与引用迁移

要求 Agent 输出引用时使用 MCP 返回的 `artifact_id`、`source_version_id`、`evidence_id`，而不是只写 `systems/xxx.md`。

推荐引用格式：

```json
{
  "artifact_id": "art_wiki_equipment",
  "source_version_id": "srcv_equipment_001",
  "evidence_id": "ev_equipment_unlock",
  "release_id": "rel_2026_06_10_001",
  "quality_score": 0.86
}
```

这会让后续 Layer 1 引用审计直接使用平台证据链。

## 9. Web 管理端范围

首期 Web 不需要做成完整知识门户，先覆盖管理闭环：

1. 项目配置：文档类型、模块 Spec、实体关系、质量规则。
2. 事实源：上传、哈希、版本、解析状态。
3. 处理任务：启动、重跑、失败诊断。
4. 审核台：查看证据、缺失、冲突、低置信度项。
5. 产物库：查看标准文档、Wiki、表 Schema、Index、Graph。
6. 发布台：质量门禁、diff、发布、回滚。
7. 查询调试：模拟 MCP 查询，查看返回证据和质量信息。

Web 调用 application service，不直接读写 `data/wiki`。

## 10. 实施阶段

### 阶段 0：基础整理

交付：

- 初始化 CodeGraph 并纳入日常架构分析。
- 建立 `kb_platform` 包结构。
- 定义领域模型和 repository port。
- 选择 SQLite 作为首期元数据存储。
- 明确 `data/sources`、`data/artifacts`、`data/releases` 目录规范。

验收：

- 可以登记项目、配置版本、事实源版本。
- 事实源导入后保留原始文件和 hash。
- 不再需要从 `data/wiki` 反推事实源。

### 阶段 1：Legacy Pipeline 适配

交付：

- 将现有 `convert/extract/tables/graph/viz` 包装为 application service 调用的 adapter。
- 每次运行生成 `ProcessingRun`。
- 产物写入 artifact store，不覆盖 published release。
- `_meta` 扩展 evidence、quality、config_version。

验收：

- 同一份事实源可以生成草稿产物。
- 草稿产物可追踪到 source version、config version、run。
- 旧命令仍可运行，但内部逐步走新服务。

### 阶段 2：质量与审核

交付：

- 文档完整性评估。
- Wiki 结构校验。
- 表 Schema 质量评估。
- Index 覆盖率校验。
- Graph 完整性校验。
- 审核决策模型。

验收：

- 不满足 P0 门禁的产物不能发布。
- 审核通过和驳回都有记录。
- 每个质量问题能定位到产物和证据。

### 阶段 3：Knowledge MCP

交付：

- `kb_*` 只读工具。
- 默认只查询 published release。
- 返回统一 envelope。
- 查询审计日志。
- `game-designer` 并行接入 MCP。

验收：

- `game-designer` 可以通过 MCP 查询 Wiki、图谱、表 Schema、证据和质量。
- Agent 不需要知道 `knowledge/wiki` 目录结构。
- MCP 查询结果包含 release、source、quality、trace。

### 阶段 4：Web 管理端

交付：

- 项目配置页。
- 事实源页。
- 任务页。
- 审核页。
- 产物页。
- 发布页。
- MCP 查询调试页。

验收：

- 项目组能在网页上维护知识标准。
- 审核和发布无需手改文件。
- 发布后 Agent 自动消费新 release。

### 阶段 5：Agent 端下线本地知识工具

交付：

- 本地 `wiki_*`、`kg_*`、`grep_search`、只读 `table_*` 改为 MCP 兼容壳或移除。
- Agent 引用使用 `evidence_id` 和 `release_id`。
- O11y 串联 MCP 查询链路。

验收：

- Agent 不再直接依赖知识库目录。
- 知识库发布新版本后，Agent 无需改代码。
- 可以追踪 Agent 使用了哪些知识、来自哪个版本、质量如何。

## 11. 架构防腐规则

必须遵守：

1. `domain` 不 import MCP、Web、LLM SDK、AgentScope。
2. MCP 工具函数不得直接读取 `data/wiki`，必须调用 query service。
3. Web API 不得绕过 application service 修改 artifact。
4. Legacy script 可以作为 adapter 存在，但不能成为新核心。
5. 发布版本不可变，回滚通过切换 active release 完成。
6. Agent session 数据不进入知识库核心模型。
7. 表写入和工作区编辑不进入默认 Knowledge MCP。

## 12. 首期最小可交付

建议首期目标压缩为一个闭环：

```text
登记事实源
 -> 解析
 -> 文档完整性评估
 -> 生成标准知识草稿
 -> 人工审核
 -> 发布
 -> Knowledge MCP 查询
 -> game-designer 使用 kb_search/kb_get_page/kb_get_evidence
```

先验证这条链路，再扩展 Wiki、Index、Graph、表知识的完整治理。

## 13. 最终验收标准

完成重构后应满足：

- 项目组通过 Web 维护知识标准。
- 所有知识产物都有来源、配置版本、处理记录、质量状态。
- Wiki、Index、Graph、Schema 可重建，且不再是唯一事实源。
- 发布版本不可变，可 diff，可回滚。
- Knowledge MCP 是 Agent 的唯一知识消费出口。
- `game-designer` 的知识查询工具统一由 Knowledge MCP 管理。
- Agent 输出可以追踪引用到 `release_id`、`artifact_id`、`source_version_id`、`evidence_id`。
